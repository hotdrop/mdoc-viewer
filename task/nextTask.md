# 次のタスク
このWebアプリのUI改善案をまとめました。これらの案をフェーズ1とフェーズ2に分けましたので、まずはフェーズ1を実行してください。
なお、フェーズ1とフェーズ2はユーザーが個別にプラン作成依頼と実装を依頼します。4つ同時に実装はしないでください。
理由は、レイアウト変更、クライアント状態、ナビゲーション、手動確認範囲が一気に増えるので、差分が大きくなります。まずは フェーズ1 を先に実装 してビューア体験を固め、その後 フェーズ2 でトップを整理する進め方が一番堅いという結論になりました。

# フェーズ1
## UI改善案
1. **ビューアの読みやすさ改善**
   - 左ナビ、本文、目次の構成は良いです。
   - ただし画面幅によって目次が下に回りやすいので、目次を右固定、またはモバイルでは「目次」ボタンで開く形にすると長文閲覧が楽になります。
2. **閲覧導線の追加**
   - ビューアに「前後のドキュメント」「このページのリンクをコピー」「検索へ戻る」を置くと、調査中の移動がかなり楽になります。
   - 編集機能は非目的なので、閲覧補助に絞るのが仕様に合っています。

## まずフェーズ1を実施する理由
どちらも /viewer/[...path]、DocumentNavigation、TableOfContents 周辺に集中していて、同じ画面のUXとして確認しやすいです。

# フェーズ2
## UI改善案
1. **トップ画面の情報密度改善**
   - 現状、ドキュメントツリーが大きく先に表示され、ロードマップやリリースノートまでスクロールが必要です。
   - トップでは「最近更新」「ロードマップ」「リリースノート」を先に見せ、ドキュメントツリーは左固定または折りたたみにすると社内ポータルとして使いやすいです。
2. **最近見たドキュメント**
   - ローカルストレージなどクライアント内だけで「最近見た」を出すと便利です。
   - サーバ側の認証・データ取得境界を崩さず実装できます。

## フェーズ2
トップを社内ポータル化する話と、クライアント内履歴をどこに見せるかが関係します。localStorage を使うならクライアントコンポーネント設計も一緒に考えた方が自然です。
